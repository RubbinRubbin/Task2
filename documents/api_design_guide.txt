API Design Guide

REST API Principles

REST (Representational State Transfer) is an architectural style for designing networked applications. A RESTful API uses HTTP methods and URLs to represent resources and operations.

Core principles of REST:
- Resources are identified by URLs (e.g., /users/123, /orders/456/items)
- HTTP methods define operations: GET (read), POST (create), PUT (update), PATCH (partial update), DELETE (remove)
- Stateless: each request contains all information needed to process it. The server does not store client session state between requests.
- Responses use standard HTTP status codes to indicate success or failure.

URL Design

Use nouns for resources, not verbs. The HTTP method already indicates the action.

Good: GET /users, POST /users, GET /users/123, DELETE /users/123
Bad: GET /getUsers, POST /createUser, GET /getUserById/123

Use plural nouns for collections: /users not /user, /orders not /order.

Nest resources to express relationships: /users/123/orders (orders belonging to user 123).

Use query parameters for filtering, sorting, and pagination: /users?role=admin&sort=created_at&page=2&limit=20

HTTP Status Codes

Use the correct status code for each response:

2xx Success:
- 200 OK: successful GET, PUT, PATCH, or DELETE
- 201 Created: successful POST that created a resource. Include Location header with the new resource URL.
- 204 No Content: successful DELETE with no response body

3xx Redirection:
- 301 Moved Permanently: resource has been permanently moved
- 304 Not Modified: cached version is still valid

4xx Client Errors:
- 400 Bad Request: malformed request syntax or invalid parameters
- 401 Unauthorized: authentication required or failed
- 403 Forbidden: authenticated but not authorized for this resource
- 404 Not Found: resource does not exist
- 409 Conflict: request conflicts with current state (e.g., duplicate email)
- 422 Unprocessable Entity: valid syntax but semantic errors (e.g., invalid email format)
- 429 Too Many Requests: rate limit exceeded

5xx Server Errors:
- 500 Internal Server Error: unexpected server failure
- 502 Bad Gateway: upstream server returned invalid response
- 503 Service Unavailable: server temporarily overloaded or in maintenance

API Versioning

Version your API to allow evolution without breaking existing clients.

Common strategies:
- URL path versioning: /api/v1/users (most common, explicit)
- Header versioning: Accept: application/vnd.myapi.v1+json (cleaner URLs but less discoverable)
- Query parameter: /users?version=1 (simple but pollutes query string)

Recommendation: use URL path versioning for public APIs. It is the most visible and easiest to understand. Maintain at most two active versions (current and previous).

Authentication and Security

Common authentication methods for APIs:
- API Keys: simple, good for server-to-server communication. Pass in header (X-API-Key) not in URL.
- OAuth 2.0: industry standard for user authorization. Use authorization code flow for web apps, client credentials for service-to-service.
- JWT (JSON Web Tokens): self-contained tokens with claims. Stateless but cannot be revoked without a blocklist.

Security best practices:
- Always use HTTPS in production
- Validate and sanitize all input
- Implement rate limiting
- Use CORS headers to control cross-origin access
- Never expose internal error details to clients
- Log all API access for auditing

Pagination

For endpoints that return collections, always implement pagination to avoid returning unbounded result sets.

Common patterns:
- Offset-based: ?page=2&limit=20 (simple but slow for large offsets)
- Cursor-based: ?cursor=abc123&limit=20 (performant, consistent with real-time data)

Include pagination metadata in responses: total count, current page, next/previous links.

Error Responses

Use a consistent error response format across all endpoints:

{
    "error": {
        "code": "VALIDATION_ERROR",
        "message": "Invalid email format",
        "details": [
            {"field": "email", "message": "Must be a valid email address"}
        ]
    }
}

Always include: a machine-readable error code, a human-readable message, and optional field-level details for validation errors.
